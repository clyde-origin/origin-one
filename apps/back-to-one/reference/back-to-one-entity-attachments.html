Back to One — Reference

Entity Attachments — image pattern

Last updated April 26, 2026 · Spec for PR #11 · Locations first, pattern reused across props, wardrobe, hmu, moodboard refs, cast reference photos

Why this pattern exists

One component, every entity surface

Same component, same storage discipline, same UI shape — different entity type. Locations is the first surface; the pattern is built to extend.

Six+ surfaces in Back to One need image attachments — locations, props, wardrobe, hmu refs, moodboard refs, cast reference photos. Building one-off image upload UIs per surface gives us six divergent components, six bucket conventions, and six places where bugs live in parallel. This pattern is one component (EntityAttachmentGallery) wired to one polymorphic table (EntityAttachment), backed by one shared bucket (entity-attachments) with structured paths.

Avatars are explicitly excluded. Crew/cast avatars are 1:1 and live as User.avatarUrl (or equivalent). They use a separate bucket and do not go through this pattern. The conceptual difference matters: an avatar is a person's identity surface; an attachment is documentation about an entity. Different things.

Schema

EntityAttachment — polymorphic, single table

Mirrors the threads polymorphic pattern. attachedToType + attachedToId identify what the image is attached to.

model EntityAttachment { id String @id @default(cuid()) attachedToType String // 'location' | 'entity-location' | 'prop' | 'wardrobe' | 'hmu' | 'moodboardRef' | 'cast' attachedToId String storagePath String // entity-attachments/{entityType}/{entityId}/{cuid}.{ext} caption String? uploadedById String? // FK to User (nullable until auth lands) uploadedAt DateTime @default(now()) width Int? height Int? mimeType String? sizeBytes Int? @@index([attachedToType, attachedToId]) @@index([uploadedById]) }

Notes for the schema PR conversation:

  • attachedToType is a string union enforced in application code, not a Postgres enum. This matches the threads polymorphic table; both can convert to enum together later if it pays off.
  • No displayOrder field in v1. Hero/cover image = most recently uploaded. If reorder becomes a real production need, add it as a dedicated PR.
  • No FK constraint on attachedToId — same reasoning as threads. Application code enforces the polymorphic relationship; batched loaders prevent N+1.
  • uploadedById nullable until auth lands. Backfill is not necessary for v1 internal users.
  • Width/height/mimeType/sizeBytes captured at upload time, not lazy-loaded. Avoids per-render image probing.
  • No soft delete in v1. Tap delete → row + Storage object both removed. If recovery becomes a need, add deletedAt as a dedicated PR.

Storage

One shared bucket, structured paths

Bucket: entity-attachments. Path convention: {entityType}/{entityId}/{randomId}.{ext}. Unsigned public URLs, no expiration. Random filenames mitigate enumeration.

Why shared bucket: 6+ surfaces would otherwise mean 6+ buckets, 6+ RLS policy sets, 6+ migration files. Shared bucket means one administrative surface. Storage paths give the same logical separation. This aligns with the polymorphic threads pattern — same architectural reasoning.

v1 sharing model: Public URLs, no expiration. Random filenames make URLs unguessable but anyone with the link can view forever. Revisit before external client beta. A leaked link to a confidential location scout is permanently leaked under v1.

Path examples:

  • entity-attachments/location/clxxx7a2.../f7a2b9c1.jpg — production-side Location row
  • entity-attachments/entity-location/clyyy8b3.../d4e5f6g7.jpg — script-side Entity(type='location') row
  • entity-attachments/prop/clzzz9c4.../a1b2c3d4.png — future Props surface

File constraints (v1): jpg, png, webp. Up to 10mb each. No video, no documents — those are future patterns and will get their own conversations.

Avatars do not use this bucket. Avatars live in a separate avatars bucket (path {userId}/{cuid}.{ext}) and resolve to User.avatarUrl. Same random-filename discipline applies.

Component contract

Props the gallery component takes

Everything it needs to render any entity surface. The same component renders on the Locations page row preview, the Locations detail sheet, and the EntityDrawer locations tab — only the type/id pair changes.

type EntityAttachmentGalleryProps = { attachedToType: 'location' | 'entity-location' | 'prop' | 'wardrobe' | 'hmu' | 'moodboardRef' | 'cast'; attachedToId: string; variant: 'row' | 'sheet' | 'lightbox'; readOnly?: boolean; // Partner role view, future maxFiles?: number; // soft cap, default unbounded for v1 onCountChange?: (n: number) => void; // for parent badge updates };

Out of scope for v1

Parked decisions

Documented here so future-you knows these were considered and deliberately deferred — not forgotten.

  • Reorder. No drag-to-reorder. Hero = most recently uploaded. Add when a real production needs it.
  • Tagging ('scout', 'reference', 'final', etc.). Captions cover the v1 use case. Revisit if free-text captions don't scale.
  • Image processing. No server-side thumbnails, compression, or face crop. Browser handles display sizing. Add when storage costs or load times demand it.
  • Soft delete / undo. Hard delete only. Add if a deletion mistake happens on a real production.
  • Permissions. No RLS in v1 — pre-auth. Wire when auth lands per #20/#21/auth cluster.
  • Signed/expiring URLs. v1 is unsigned public URLs. Revisit before external client beta. Documented in DECISIONS.md
  • Error states (upload fail, network, file too big, wrong type, image fails to load). Standard handling — toast + retry. Not drafted as a separate UI surface.
  • Avatars. Explicitly not this pattern. User.avatarUrl column + separate avatars bucket. See #20/#21.
